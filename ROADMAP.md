# ROADMAP

这份路线图用于控制 NSJ Lock 的小步迭代。它不替代 `AGENTS.md` 的协作规则，也不替代 `docs\release.md` 的发布步骤。

## 当前基线

当前产品基线是一个本地优先的 Windows x64 WPF 桌面工具：

- 保护开启后，可以使用固定锁定模式把系统主音量拉回目标百分比，也可以使用动态限制模式在输出峰值过高时临时压低系统主音量。
- 设置保存在本地 JSON 文件。
- 主窗口关闭后进入托盘，托盘菜单提供保护开关和退出。
- 支持主题切换和中英文界面。
- 构建、测试、发布优先使用仓库本地 `.dotnet` SDK 和仓库脚本。

## 近期优先级

### 1. 文档对齐

目标：让仓库文档和当前发布流程保持一致。

范围：

- 对齐 README 和 release 文档中的版本示例。
- 明确哪些文档属于私有协作上下文，哪些会进入公开导出。
- 保持 `AGENTS.md`、`PROJECT.md`、`ROADMAP.md`、`README.md` 的职责不重叠。

验证：

- `git diff --check`
- 人工检查版本示例是否和 `src\NSJLock.App\NSJLock.App.csproj` 的版本策略一致。
- 如修改公开导出规则，检查导出树中没有内部文件或构建产物。

### 2. 核心保护稳定性

目标：确保固定锁定和动态限制行为简单、可预测、可测试。

范围：

- 固定锁定模式继续保持“低于目标拉高、高于目标拉低”的硬目标语义。
- 动态限制模式允许用户手动调系统音量，只在输出峰值超过阈值时临时压低。
- 动态限制模式在峰值安全后逐步恢复到用户目标音量。
- 保持默认输出设备变化时的行为可理解。
- 补足对边界值、设备不可用、峰值不可读、设置加载失败的测试。

验证：

- `.\.dotnet\dotnet.exe build .\NSJLock.sln`
- `.\.dotnet\dotnet.exe test .\NSJLock.sln`
- 必要时手动运行应用，观察实际系统音量变化。

### 3. 桌面体验打磨

目标：让日常使用更稳、更清楚，但不扩大产品范围。

范围：

- 保持顶部保护开关可见。
- 优化状态文字、错误提示和设备名称显示。
- 检查主题和语言切换是否稳定保存。
- 检查托盘显示、隐藏、退出路径是否一致。

验证：

- ViewModel 测试覆盖状态和文本变化。
- WPF Debug 运行检查主要窗口、托盘和退出路径。
- 不修改音频控制主链路，除非任务明确要求。

### 4. 发布收尾标准化

目标：每次发布都能确认真实交付物，而不只确认构建通过。

范围：

- 发布前递增项目版本字段。
- 运行 `scripts\publish-single-file.ps1`。
- 验证 `NSJLock.exe` 和当前版本号 exe 都已生成。
- 验证文件版本信息正确。
- 确认发布产物没有进入 Git。

验证：

- `.\.dotnet\dotnet.exe test .\NSJLock.sln`
- `.\.dotnet\dotnet.exe build .\NSJLock.sln`
- 发布脚本输出检查。
- 文件版本和发布目录检查。
- `git status --short --ignored`

## 暂不做

- 单应用音量控制。
- 虚拟声卡、底层音频流 DSP、多段压缩或复杂音频处理。
- 参会者级别的单独音量控制。
- 会议软件专用模式或 API 集成。
- 云同步、账号系统、远程配置。
- 自动更新器。
- 大规模 UI 重写。

## 决策门槛

- 涉及音频控制、权限、文件系统或发布流程的改动视为高风险，必须先确认方案。
- 涉及多个模块的改动优先拆成多个可审查的小 diff。
- 新功能必须能说明用户场景、失败模式和验证方式。
- 如果只是文案、说明或开发流程变化，优先走文档小 diff。

## 完成定义

一个迭代只有在以下条件满足后才算完成：

- 改动范围清楚，未顺手扩大。
- 至少运行了与改动匹配的验证。
- 失败或 warning 被记录，没有被吞掉。
- 交付说明写清楚改了什么、验证了什么、还剩什么风险。
